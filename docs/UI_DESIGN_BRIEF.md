# GameBot UI 设计需求文档

## 一、项目背景

### 这是什么

一个 **Windows 桌面应用**（WebUI 嵌入 PyWebView 窗口，最终打包成 .exe），用于自动化操控安卓模拟器中的手游（和平精英 / PUBG Mobile）。

### 核心业务

用户在 Windows 电脑上开 6~12 个雷电模拟器（LDPlayer），每个模拟器里跑一个游戏账号。这些账号分成 **2 队**（A 队 / B 队），每队 3 个机器人账号 + 2 个真人玩家 = 5 人组队。两队同时进入匹配，目标是**匹配到同一场对局**，匹配成功后机器人断网退出，真人留在对局中继续游戏。如此循环。
å
### 用户画像

- 非技术人员（网吧老板、代练工作室）
- 需要 7×24 小时无人值守运行
- 可能同时管理多台 Windows 电脑
- 中文界面

---

## 二、应用架构

```
┌──────────────────────────────────────┐
│         PyWebView 桌面窗口            │
│         (1280×800 默认大小)            │
│                                      │
│    ┌──────────────────────────────┐   │
│    │        WebUI (React)         │   │
│    │    ← WebSocket 实时通信 →     │   │
│    │        FastAPI 后端           │   │
│    │          ↓ ADB 命令           │   │
│    │    雷电模拟器 ×6~12 个         │   │
│    └──────────────────────────────┘   │
└──────────────────────────────────────┘
```

- 后端通过 ADB（Android Debug Bridge）控制每个模拟器：截图、点击、启动/停止应用
- 前端通过 WebSocket 接收实时状态更新（每秒推送一次）
- 前端通过 REST API 发送控制指令（启动、停止、配置）

## 三、页面结构

整个应用只有 **一个页面**，通过顶部导航切换不同视图。不需要路由。

### 3.1 全局布局

```
┌─ Header（固定顶部）──────────────────────────────────────────┐
│ Logo | 导航(控制面板/设置) | 状态指标 | 操作按钮 | 日志开关   │
├──────────────────────────────────────┬───────────────────────┤
│                                      │                       │
│            主内容区                    │     日志侧边栏        │
│         (随导航切换)                   │    (可收起/展开)       │
│                                      │                       │
└──────────────────────────────────────┴───────────────────────┘
```

### 3.2 窗口规格

- 默认大小：1280×800
- 最小大小：1024×600
- 深色主题（7×24 运行不刺眼）
- 需要适配不同 Windows 分辨率（1080p / 1440p / 4K）

## 四、Header（顶部栏）

固定在页面最顶部，所有视图共享。

### 包含元素（从左到右）

1. **Logo + 应用名称**
   - Logo：小图标（32×32）
   - 名称："GameBot" 或自定义
   
2. **导航按钮**（2 个）
   - 控制面板（默认选中）
   - 设置

3. **实时状态指标**（运行时显示，未运行时隐藏）
   - "运行中 4/6" — 正在执行任务的实例数 / 总数
   - "已就绪 2/6" — 已完成任务到达大厅的实例数
   - "出错 0" — 出错的实例数（有错误时红色高亮）
   - 运行时长计时器 "12m30s"

4. **操作按钮**
   - 未运行时：「启动全部」（主按钮，醒目）
   - 运行中时：「停止」（红色/危险色）

5. **日志开关**
   - "显示日志" / "隐藏日志" 切换右侧日志面板

## 五、控制面板（主视图）

控制面板有两个状态：**未运行**和**运行中**，展示完全不同的内容。

### 5.1 未运行状态 — 部署视图

显示本机检测到的所有模拟器实例，让用户快速了解当前环境状态并开始测试。

#### 数据来源

后端自动检测本机安装的雷电模拟器实例，返回：
```json
[
  {"index": 0, "name": "雷电模拟器", "running": true, "adb_serial": "emulator-5554"},
  {"index": 1, "name": "雷电模拟器-1", "running": false, "adb_serial": "emulator-5556"},
  ...
]
```

#### 页面布局

```
部署中心
6 个模拟器 · 4 个在线                                    [刷新]

● 在线 · 可操作
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ #0 雷电模拟器 │ │ #1 雷电模拟器 │ │ #2 雷电模拟器 │
│ emulator-5554│ │ emulator-5556│ │ emulator-5558│
│ ● 在线       │ │ ● 在线       │ │ ● 在线       │
│ [▶ 测试运行]  │ │ [▶ 测试运行]  │ │ [▶ 测试运行]  │
└──────────────┘ └──────────────┘ └──────────────┘

○ 离线 · 需在雷电多开器中启动
┌──────────────┐ ┌──────────────┐

│ 离线          │ │ 离线          │
└──────────────┘ └──────────────┘

提示：点击「测试运行」单独测试一个实例的完整流程。
全部准备好后，在「设置」中配置分组，再用「启动全部」批量运行。
```

#### 交互说明

- **在线实例卡片**：显示编号、名称、ADB 地址、在线状态、「测试运行」按钮
- **离线实例卡片**：灰色/半透明，提示需要先启动模拟器
- **「测试运行」按钮**：点击后对该实例执行完整自动化流程（加速器→游戏→弹窗清理→大厅），点击后页面自动切换到运行中状态
- **「刷新」按钮**：重新检测模拟器状态（自动每 5 秒刷新一次，也可手动刷新）
- 卡片排列：3 列网格，响应式

### 5.2 运行中状态 — 作战视图

这是核心视图，用户 90% 的时间都在看这个页面。需要给用户**强烈的掌控感**。

#### 页面结构

```
┌─ 全局流水线进度 ──────────────────────────────────────────┐
│ ⚡加速器 6/6 → 🎮启动游戏 4/6 → 🧹清理弹窗 2/6 → 🏠大厅 1/6  │
│ [████████████████] [█████████░░░] [████░░░░░░░] [██░░░░░░] │
│                                              运行 12m30s    │
└──────────────────────────────────────────────────────────────┘

┌─── A 队 ──────────────────────┐ ┌─── B 队 ──────────────────────┐
│                                │ │                                │
│ ★ 队长卡片（大，带实时截图）    │ │ ★ 队长卡片（大，带实时截图）    │
│                                │ │                                │
│ 队员卡片 ×2（小，可展开截图）   │ │ 队员卡片 ×2（小，可展开截图）   │
│                                │ │                                │
│ 真人 1: 待加入  真人 2: 待加入  │ │ 真人 1: 待加入  真人 2: 待加入  │
│                                │ │                                │
└────────────────────────────────┘ └────────────────────────────────┘
```

#### 5.2.1 全局流水线进度

一条横向的 4 段流水线，展示所有实例的整体进展：

| 阶段 | 图标 | 说明 |
|------|------|------|
| 加速器 | ⚡ | 启动 VPN 加速器并验证网络 |
| 启动游戏 | 🎮 | 启动游戏 APP 并等待加载完成 |
| 清理弹窗 | 🧹 | 自动关闭游戏内弹出的各种活动/公告弹窗 |
| 进入大厅 | 🏠 | 到达游戏大厅，准备组队 |

每段显示：
- 图标 + 阶段名
- 完成数/总数（如 "4/6"）
- 进度条（按完成比例填充）
- 全部完成时进度条变为绿色

右侧显示总运行时长。

#### 5.2.2 队伍面板（A 队 / B 队）

左右两栏并排，每栏代表一支队伍。

**队伍标题栏**：
- 彩色竖条标识（A 队蓝色、B 队橙色）
- "A 队" / "B 队"
- "3 机器人 + 2 真人"
- 全员就绪时显示「全员就绪」标签

**队长卡片**（每队 1 个，大卡片）：
```
┌─────────────────────────────────────────┐
│ [实时截图 180×120]  │ #0 ★队长  清理弹窗  │
│                     │ ⚡✓ 🎮✓ 🧹→ 🏠○     │
│                     │ emulator-5554  12s  │
└─────────────────────────────────────────┘
```
- 左侧：实时截图缩略图（每 3 秒自动刷新），固定可见不需要点击展开
- 右侧上：实例编号 + ★队长标签 + 昵称 + 当前状态文字
- 右侧中：4 步进度图标（已完成✓ / 进行中动画 / 未开始灰色）
- 右侧下：ADB 地址 + 当前阶段耗时

**队员卡片**（每队 2 个，小卡片，2 列排列）：
```
┌────────────────────┐
│ #1 队员  启动游戏    │
│ ████████░░░░░░░░░░ │ ← 4段迷你进度条
└────────────────────┘
```
- 编号 + 昵称 + 当前状态
- 迷你进度条（4 段，颜色填充表示进度）
- 点击可展开显示实时截图
- 错误时显示红色错误信息

**真人占位**（每队 2 个）：
```
┌─ 真人 1: 待加入 ─┐  ┌─ 真人 2: 待加入 ─┐
└──────────────────┘  └──────────────────┘
```
- 虚线边框，表示占位符
- 状态：待加入 / 已加入 / 已准备 / 游戏中
- 后续功能会扩展为显示真人的加入状态

#### 5.2.3 各实例的状态列表

实例在运行过程中会经历以下状态：

| 状态值 | 中文标签 | 颜色 | 说明 |
|--------|---------|------|------|
| init | 等待启动 | 灰色 | 初始状态 |
| accelerator | 启动加速器 | 蓝色 | 正在启动 VPN 加速器 |
| launch_game | 启动游戏 | 蓝色 | 正在启动游戏并等待加载 |
| wait_login | 等待登录 | 蓝色 | 等待自动登录完成 |
| dismiss_popups | 清理弹窗 | 黄色/动画 | 正在自动关闭弹窗 |
| lobby | 大厅就绪 | 绿色 | 已到达游戏大厅 |
| map_setup | 设置地图 | 紫色 | 队长正在选择游戏模式和地图 |
| team_create | 创建队伍 | 紫色 | 队长正在创建组队 |
| team_join | 加入队伍 | 紫色 | 队员正在加入队伍 |
| done | 完成 | 绿色 | 任务完成 |
| error | 出错 | 红色 | 出现错误，显示错误信息 |

> 状态为蓝色/黄色时应有呼吸灯或脉冲动画，表示"正在工作中"。

## 六、设置页

设置页包含 3 个区域，纵向排列。

### 6.1 环境配置

基础路径配置，4 个输入框，2×2 网格排列。

| 字段 | 类型 | 提示文字 | 说明 |
|------|------|---------|------|
| 雷电模拟器路径 | 文本输入 | D:\leidian\LDPlayer9 | LDPlayer 安装目录 |
| ADB 路径 | 文本输入 | 留空自动使用模拟器自带 | 可选，留空则自动拼接 |
| 游戏包名 | 文本输入 | com.tencent.tmgp.pubgmhd | 一般不需要改 |
| 目标地图 | 文本输入 | 狙击团竞 | 队长自动选择的地图名 |

底部一个「保存设置」按钮。

### 6.2 模拟器检测

自动检测本机雷电模拟器实例的区域。

```
模拟器检测  自动发现本机雷电模拟器实例

[刷新检测]  [同步到实例分配]     检测到 6 个模拟器

┌──────────┐ ┌──────────┐ ┌──────────┐
│ #0 雷电   │ │ #1 雷电-1 │ │ #2 雷电-2 │
│ ● 运行中  │ │ ● 运行中  │ │ ○ 已关闭  │
│ :5554    │ │ :5556    │ │ :5558    │
└──────────┘ └──────────┘ └──────────┘
...
```

- **「刷新检测」按钮**：重新扫描模拟器
- **「同步到实例分配」按钮**：把检测到的模拟器自动填入下方的实例分配表
- 每个模拟器小卡片显示：编号、名称、在线/离线状态、ADB 端口

### 6.3 实例分配表

一个可编辑的表格，用于将模拟器实例分配到队伍中。

| 列 | 类型 | 说明 |
|----|------|------|
| 模拟器 | 只读 | 编号 + 检测到的模拟器名称 |
| 状态 | 只读 | 在线/离线标签 |
| ADB | 只读 | 端口号（自动计算） |
| 组 | 下拉框 | A / B |
| 角色 | 下拉框 | 队长 / 队员 |
| 昵称 | 文本输入 | 用户自定义，默认为模拟器名称 |
| 游戏ID | 文本输入 | 用于后续校验玩家身份 |
| 删除 | 按钮 | × 图标，点击移除该行 |

表格底部：
- 「+ 添加实例」按钮
- 「保存分配」按钮

**提示信息框**：
> 实例编号对应雷电多开器中的模拟器序号（从 0 开始）。每组需要 1 个队长 + 2 个队员。ADB 端口自动计算。

### 布局

```
┌─ 日志  14 条 ──────── 清空  关闭 ─┐
├─ 过滤: [全部] [0] [1] [2] [3] ... ─┤
│                                     │
│ ● 22:39:21 [2] 启动加速器           │
│ ● 22:39:23 [2] 加速器已连接 ✓       │
│ ● 22:39:25 [2] 网络验证通过 ✓       │
│ ● 22:39:26 [2] 启动游戏             │
│ ● 22:39:45 [SYS] 游戏加载完成       │
│ ⚠ 22:39:46 [2] 清理弹窗中...        │
│ ● 22:39:47 [守卫] 自动清除弹窗       │
│ ✓ 22:39:52 [2] 到达大厅             │
│                                     │
│                         ← 自动滚动   │
└─────────────────────────────────────┘
```

### 元素说明

- **标题栏**：日志图标 + "日志" + 条数 + 清空按钮 + 关闭按钮
- **过滤栏**：按钮组，可按实例编号过滤日志（全部 / 0 / 1 / 2 / 3 / 4 / 5）

- **宽度**：约 380px
- **可收起**：点击 Header 中的"隐藏日志"完全隐藏，主内容区占满宽度

## 八、数据模型

### 8.1 WebSocket 推送消息类型

前端通过 WebSocket 接收 3 种消息：

```typescript
// 1. 全量快照（每秒推送一次）
{
  type: "snapshot",
  instances: {
    "0": { index: 0, group: "A", role: "captain", state: "dismiss_popups", 
           nickname: "玩家1", error: "", state_duration: 12.3 },
    "1": { index: 1, group: "A", role: "member", state: "lobby", ... },
    ...
  },
  stats: { total_attempts: 0, success_count: 2, error_count: 0, running_duration: 125.6 }
}

// 2. 状态变化（实例状态切换时推送）
{
  type: "state_change",
  data: { instance: 0, old: "accelerator", new: "launch_game" }
}

// 3. 日志（实时推送）
{
  type: "log",
  data: { timestamp: 1712736000.5, instance: 0, level: "info", 
          message: "加速器已连接 ✓", state: "accelerator" }
}
```

### 8.2 REST API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/emulators | 检测本机模拟器实例列表 |
| POST | /api/start | 启动所有已配置的实例 |
| POST | /api/start/{idx} | 启动单个实例（测试用） |
| POST | /api/stop | 停止所有实例 |
| GET | /api/status | 获取当前所有实例状态 |
| GET | /api/settings | 获取设置 |
| PUT | /api/settings | 保存设置 |
| GET | /api/accounts | 获取实例分配列表 |
| PUT | /api/accounts | 保存实例分配 |
| GET | /api/screenshot/{idx} | 获取指定实例的实时截图（JPEG） |
| GET | /api/health | 健康检查 |

## 九、交互流程

### 9.1 首次使用流程

```
用户打开应用
  → 控制面板显示"部署中心"
  → 未检测到模拟器（路径不对）
  → 用户点击"设置"
  → 填写雷电模拟器路径 → 保存
  → 回到控制面板 → 刷新 → 检测到 6 个模拟器
  → 用户点击"设置" → "同步到实例分配" → 自动分组 → 保存
  → 回到控制面板 → 点击某个在线实例的"测试运行"
  → 页面切换到运行中视图 → 观察流程
  → 测试通过 → 停止 → 点击"启动全部"正式运行
```

### 9.2 日常运行流程

```
用户打开应用
  → 控制面板显示已检测到的模拟器
  → 点击"启动全部"
  → 作战视图：全局进度条 + 两队面板
  → 观察每个实例的进度，队长卡片显示实时截图
  → 全部到达大厅 → 后续会自动组队、匹配（待实现）
  → 遇到问题查看右侧日志
  → 需要停止时点击"停止"
```

### 9.3 错误处理

- 实例出错时：卡片变红，显示错误信息
- 日志中会显示红色错误条目
- Header 的状态指标会显示"出错 1"（红色）

## 十、设计风格要求

### 10.1 基调

- **深色主题**（必须），背景色 #0a0e1a 级别的深蓝黑
- 7×24 小时长时间观看不刺眼
- **科技感 / 控制中心风格**，类似任务控制台（mission control）
- 不要卡通、不要可爱、不要圆润过头

### 10.2 视觉元素

- **玻璃拟态**（Glassmorphism）：卡片半透明 + 模糊背景
- **渐变点缀**：按钮、进度条、Logo 可使用蓝紫渐变
- **状态发光**：在线/运行中的指示灯带微弱光晕（glow）
- **微动画**：
  - 进行中的状态有呼吸灯效果
  - 页面切换有滑入动画
  - 进度条有平滑过渡
- 卡片边框在 hover 时有微弱的渐变边框效果

### 10.3 颜色系统

| 用途 | 颜色 | 场景 |
|------|------|------|

## 十一、技术约束

- **技术栈**：React 19 + TypeScript + Tailwind CSS 4 + Zustand
- **构建工具**：Vite 8
- **无需路由**：单页应用，组件切换
- **不要引入重型 UI 库**（如 Ant Design、Material UI），纯 Tailwind + 手写组件
- **WebSocket 自动重连**（断线 3 秒后自动重连）
- **响应式**：主要适配 1280×800，兼容 1920×1080 和更大屏幕
- 最终打包为 Windows .exe（通过 PyWebView 嵌入），不需要考虑移动端

## 十三、后续扩展（设计时预留空间）

以下功能当前不实现，但 UI 设计时需要预留扩展余地：

1. **组队阶段**：队长创建队伍、分发口令码、队员加入 → 在"大厅就绪"之后会增加新的阶段
2. **匹配同步**：两队同时点击开始匹配 → 全局流水线会增加"组队"→"匹配"→"验证"阶段
3. **匹配统计**：成功/失败/重试次数 → Header 状态指标会扩展
4. **多机管理**：一台电脑管理多台 Windows 机器上的模拟器 → 可能需要"机器"层级
5. **12 实例扩展**：最多 12 个实例，4 队 → TeamPanel 需要能适配更多队伍
6. **告警通知**：封号、掉线等严重异常 → 可能需要通知弹窗或声音提醒
